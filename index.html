1. Entorno de desarrollo del software

En este apartado reviso los elementos clave que conforman el entorno de desarrollo y por qué importan.

• Introducción

El entorno de desarrollo describe las herramientas, procesos, metodologías y la infraestructura que facilitan crear software. Incluye IDEs, control de versiones, servidores, dependencias y políticas de despliegue.

• Entendimiento del Negocio

Antes de escribir código, es esencial entender las necesidades del negocio: objetivos estratégicos, métricas clave, restricciones regulatorias y el perfil del usuario final. Esto permite priorizar funcionalidades útiles y evitar desperdicio de esfuerzo.

Actividades típicas:
  • Entrevistas con stakeholders
  • Mapas de proceso
  • Definición de KPIs

• Mejoramiento del proceso actual

Evaluar el flujo de trabajo actual para detectar cuellos de botella: tiempos de despliegue largos, pruebas manuales o falta de retroalimentación. Usar métricas y un enfoque iterativo (por ejemplo, Kanban o Scrum) para mejorar continuamente.

Ejemplo: Implementar CI/CD para reducir el tiempo entre la creación de una funcionalidad y su entrega en producción.

2. El Modelo del Negocio

Modelar el negocio ayuda a trasladar las necesidades empresariales a un sistema coherente y mantenible.

• Modelo del Dominio

Describe las entidades, relaciones y reglas del negocio en un lenguaje cercano al dominio. Suele representarse con diagramas de clases o entidades (ERD).

  • Entidades: Cliente, Producto, Pedido
  • Reglas: Validez de estados, límites, políticas

• Modelo de Casos de Uso del Negocio

Los casos de uso describen cómo los actores (usuarios, sistemas externos) interactúan con el sistema para alcanzar objetivos del negocio. Permiten identificar requisitos funcionales claros.

• Modelo de Objetos del Negocio

Traducción técnica del modelo del dominio a objetos/entidades en el sistema: atributos, métodos y límites de responsabilidad. Aquí se decide qué lógica vive en el dominio vs servicios.

• Representación de un Modelo de Negocio – Ejemplos Prácticos

A continuación hay dos ejemplos breves y aplicables:

  1. Comercio electrónico: Entidades — Usuario, Carrito, Producto, Inventario, Pago. Flujo — Usuario agrega al carrito → crea orden → procesa pago → actualiza inventario.
  2. Gestión de proveedores: Entidades — Proveedor, Contrato, Factura, Pago. Flujo — Proveedor envía factura → validación → aprobación → pago.

• Pasando del Modelo del Negocio al Modelo del Sistema

Se realiza una descomposición técnica: servicios, APIs, bases de datos, colas, componentes frontend y responsabilidades. Esto incluye decisiones arquitectónicas (monolito vs microservicios), patrones (CQRS, Event Sourcing) y criterios non-functional (seguridad, escalabilidad).

Plantilla práctica rápida

Usa este checklist para pasar del negocio al sistema:

  • 1. Mapear entidades del dominio.
  • 2. Definir casos de uso prioritarios.
  • 3. Diseñar interfaces (API / UI) para cada caso de uso.
  • 4. Selección de tecnologías (DB, runtime, frameworks).
  • 5. Definir pruebas automatizadas y pipelines.

Ejemplos prácticos y recursos

Material de apoyo y una pequeña guía para actividades que puedes usar en clase o para tu proyecto.

Actividades sugeridas

  1. Realizar un diagrama de dominio para un proceso real de tu trabajo o estudio.
  2. Escribir 3 casos de uso con pasos desde el actor hasta el resultado final (pre y post condiciones).
  3. Construir un modelo de objetos en UML y mapearlo a tablas de base de datos.

Sugerencias de despliegue

  • GitHub Pages o Netlify para hosting estático.
  • Integrar un flujo básico de CI que valide HTML/CSS y ejecute tests de accesibilidad.
